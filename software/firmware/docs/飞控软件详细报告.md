## 1. 系统架构总览

本飞控系统基于 **ESP32-S3 双核处理器**，使用 **ESP-IDF** 框架开发，核心算法移植自开源飞控项目 **Crazyflie**（Bitcraze AB），并进行了大量硬件适配和算法优化。

### 1.1 硬件平台

| 组件 | 型号/规格 |
|------|----------|
| 主控芯片 | ESP32-S3 (双核 Xtensa LX7, 240MHz) |
| IMU传感器 | BMI088 (SPI接口) - 六轴加速度计+陀螺仪 |
| 气压计 | MS5611 (I2C接口) |
| 光流/测距 | MTF01 (UART接口) - ToF + 光流 |
| 遥控接收 | SBUS协议 (反相UART, 100kbps) |
| 电机驱动 | MCPWM → 无刷ESC (400Hz, 1000-2000μs) |
| 通信 | Wi-Fi (AP/STA模式) + CRTP协议 |

### 1.2 构型定义

系统采用 **X构型四旋翼**（通过 `config.h` 中 `#define QUAD_FORMATION_X` 定义）：

```
    M2(CCW)   M1(CW)
       \     /
        \   /
         \ /
          X     ← 机头朝前
         / \
        /   \
       /     \
    M3(CW)   M4(CCW)
```

## 3. 系统启动流程

系统启动由 `app_main()` → `systemLaunch()` → `systemTask()` 三级调用链完成：

### 3.1 app_main() - 入口函数

```c
void app_main()
{
    // 1. NVS Flash 初始化（为Wi-Fi做准备）
    nvs_flash_init();

    // 2. 平台初始化
    platformInit();

    // 3. BMI088 硬件自检（SPI读取验证）
    spiDrvInit(SPI_DRV_HOST_DEFAULT);
    bmi088_init_with_default_config(&bmi088_dev);
    // 读取10次陀螺仪+加速度计数据验证

    // 4. SBUS 硬件自检
    sbusInit();
    // 尝试读取10帧SBUS数据验证

    // 5. 启动系统任务
    systemLaunch();
}
```

### 3.2 systemTask() - 系统初始化序列

```c
void systemTask(void *arg)
{
    // === 底层外设 ===
    ledInit();
    wifiInit();

    // === 高级模块 ===
    systemInit();        // NVS配置、ADC、LED动画、电源管理、蜂鸣器
    commInit();          // CRTP通信栈
    commanderInit();     // 指令管理器（含优先级仲裁）
    motorTestInit();     // 电机测试模块
    extRxInit();         // SBUS外部接收器

    // === 状态估计器 ===
    estimatorKalmanTaskInit();  // 卡尔曼滤波任务初始化

    // === 核心控制 ===
    stabilizerInit(kalmanEstimator);  // 稳定器初始化（含传感器、控制器、电机）
    vehicleStateInit();               // PX4风格状态机

    // === 自检 ===
    // 逐一测试各模块，全部通过后调用 systemStart()
    systemStart();  // 释放canStartMutex，允许各任务开始运行
}
```

**关键设计**: `canStartMutex` 互斥信号量在 `systemInit()` 中被获取，阻止所有等待 `systemWaitStart()` 的任务启动。只有当所有模块自检通过后，`systemStart()` 释放该互斥量，各任务才开始真正运行。

## 4. 稳定器主循环 (Stabilizer)

稳定器是整个飞控的核心调度引擎，以 **1kHz** 频率运行（由BMI088陀螺仪中断驱动）。

### 4.1 主循环流程

```c
static void stabilizerTask(void *param)
{
    systemWaitStart();          // 等待系统初始化完成
    // 等待传感器校准...

    while (1) {
        sensorsWaitDataReady();  // 阻塞等待IMU中断（1kHz节拍源）

        // ① 状态估计
        stateEstimator(&state, &sensorData, &control, tick);

        // ② 获取目标设定值
        commanderGetSetpoint(&setpoint, &state);

        // ③ 情景感知更新
        sitAwUpdateSetpoint(&setpoint, &sensorData, &state);

        // ④ 控制器计算
        controller(&control, &setpoint, &sensorData, &state, tick);

        // ⑤ 飞行器状态更新 (100Hz)
        vehicleStateUpdate();

        // ⑥ 电机输出
        if (vehicleIsArmed()) {
            powerDistribution(&control);    // 混控分配
        } else {
            powerStop();                    // 停机
        }

        // ⑦ 看门狗喂狗
        esp_task_wdt_reset();
    }
}
```

### 4.2 关键数据结构

| 变量 | 类型 | 说明 |
|------|------|------|
| `sensorData` | `sensorData_t` | 传感器原始数据（IMU/气压计/磁力计） |
| `state` | `state_t` | 当前状态估计（姿态/位置/速度） |
| `setpoint` | `setpoint_t` | 目标设定点（来自遥控/上位机） |
| `control` | `control_t` | 控制器输出（roll/pitch/yaw/thrust） |

### 4.3 硬件看门狗

```c
#define STABILIZER_WDT_TIMEOUT_SEC 3  // 3秒超时
esp_task_wdt_config_t twdt_config = {
    .timeout_ms = 3000,
    .trigger_panic = true,  // 超时触发系统重启
};
```

如果稳定器任务在3秒内未能喂狗（如死循环或阻塞），ESP32会自动重启，防止失控。

## 5. 传感器驱动与数据采集

### 5.1 BMI088 IMU (SPI)

BMI088是博世的六轴惯性传感器，包含三轴加速度计和三轴陀螺仪，通过SPI总线通信。

**关键配置参数**:
- 陀螺仪量程: ±2000°/s, 灵敏度: 0.061°/s/LSB
- 加速度计量程: ±6g, 灵敏度: 0.000183g/LSB
- 采样率: 1kHz (中断驱动)

**坐标轴映射** (BMI088物理轴 → 飞控体坐标系):

```c
// sensors_bmi088_spi_ms5611.c 中的坐标映射
sensorData.gyro.x = -(gyroRaw.y - gyroBias.y) * DEG_PER_LSB;  // 飞控X ← -BMI088.Y
sensorData.gyro.y =  (gyroRaw.x - gyroBias.x) * DEG_PER_LSB;  // 飞控Y ←  BMI088.X
sensorData.gyro.z = -(gyroRaw.z - gyroBias.z) * DEG_PER_LSB;  // 飞控Z ← -BMI088.Z

accScaled.x = -accelRaw.y * G_PER_LSB / accScale;  // 同上映射
accScaled.y =  accelRaw.x * G_PER_LSB / accScale;
accScaled.z =  accelRaw.z * G_PER_LSB / accScale;
```

**中断驱动机制**:

BMI088的加速度计和陀螺仪各有独立的数据就绪中断引脚(INT1/INT3)。两个中断共享同一个二进制信号量 `sensorsDataReady`：

```c
// 中断处理流程
ACC/GYRO INT → ISR → 设置dataReady标志 → 给出sensorsDataReady信号量
    → sensorsTask被唤醒 → 检查标志决定读取哪个传感器
    → SPI读取原始数据 → 去偏 → 坐标映射 → 低通滤波 → 推入队列
```
### 5.2 陀螺仪校准

#### 静态校准（上电校准）

系统启动时采集 1024 个静止样本，计算均值作为零偏补偿值：

```c
#define SENSORS_NBR_OF_BIAS_SAMPLES 1024

static bool processGyroBias(int16_t gx, int16_t gy, int16_t gz, Axis3f *gyroBiasOut)
{
    // 1. 将样本加入环形缓冲区
    sensorsAddBiasValue(&gyroBiasRunning, gx, gy, gz);
    
    // 2. 缓冲区满后计算方差和均值
    sensorsCalculateVarianceAndMean(&gyroBiasRunning, &variance, &mean);
    
    // 3. 方差低于阈值 → 认为静止 → 均值即为零偏
    if (variance.x < GYRO_VARIANCE_THRESHOLD_X &&
        variance.y < GYRO_VARIANCE_THRESHOLD_Y &&
        variance.z < GYRO_VARIANCE_THRESHOLD_Z) {
        gyroBiasOut->x = mean.x;
        gyroBiasOut->y = mean.y;
        gyroBiasOut->z = mean.z;
        return true;  // 校准完成
    }
}
```

#### 动态校准（飞行中校准）

着陆后自动重新校准零偏，补偿温漂引起的漂移：

```c
#define GYRO_DYNAMIC_CALIB_DELAY_MS  2000   // 着陆后2秒开始
#define GYRO_DYNAMIC_CALIB_ALPHA     0.002  // 低通滤波系数 (~1s时间常数)

static void gyroDynamicCalibUpdate(int16_t gx, int16_t gy, int16_t gz, bool quadIsFlying)
{
    if (!quadIsFlying) {
        // 着陆后经过延迟，使用指数移动平均更新零偏
        gyroBias.x += alpha * (gx_raw - gyroBias.x);
        gyroBias.y += alpha * (gy_raw - gyroBias.y);
        gyroBias.z += alpha * (gz_raw - gyroBias.z);
    }
}
```

### 5.3 数字滤波

所有传感器数据经过 **二阶低通滤波器 (Biquad LPF)** 处理：

| 传感器 | 截止频率 | 采样率 |
|--------|---------|--------|
| 陀螺仪 | 30 Hz | 1000 Hz |
| 加速度计 | 20 Hz | 1000 Hz |

### 5.4 MTF01 光流/测距模块

MTF01通过UART输出光流速度和ToF距离数据，经卡尔曼滤波器融合后用于位置估计：

- **ToF距离**: 10-10000mm, 标准差 0.05m
- **光流速度**: cm/s @ 1m高度, 标准差 0.05
- **死区滤波**: 低于20 cm/s视为静止
- **低通滤波**: α=0.3 平滑光流输出

## 6. 姿态解算算法

系统实现了两套独立的姿态解算方案，通过 `system.c` 中的 `StateEstimatorType estimator = kalmanEstimator` 选择使用哪套方案。**默认使用卡尔曼滤波估计器**，其姿态解算集成在第7章所述的EKF中，通过陀螺仪积分预测姿态并由加速度计周期性校正；互补滤波估计器则使用本章描述的 Mahony/Madgwick 算法作为替代。

### 6.1 Mahony AHRS 互补滤波（互补估计器模式使用）

该算法位于 `sensfusion6.c`，由互补估计器 (`estimator_complementary.c`) 调用，**不是当前默认路径**。Mahony算法使用PI控制器将加速度计的重力方向估计与陀螺仪积分进行融合。

**算法核心步骤**:

```c
// 步骤1: 归一化加速度计测量值
recipNorm = invSqrt(ax*ax + ay*ay + az*az);
ax *= recipNorm; ay *= recipNorm; az *= recipNorm;

// 步骤2: 利用当前四元数估计重力方向
halfvx = qx*qz - qw*qy;
halfvy = qw*qx + qy*qz;
halfvz = qw*qw - 0.5f + qz*qz;

// 步骤3: 计算误差（估计重力方向 × 测量重力方向）
halfex = (ay*halfvz - az*halfvy);    // 叉积得到姿态误差
halfey = (az*halfvx - ax*halfvz);
halfez = (ax*halfvy - ay*halfvx);

// 步骤4: PI反馈校正
if (twoKi > 0.0f) {
    integralFBx += twoKi * halfex * dt;  // 积分项（消除陀螺零偏）
    integralFBy += twoKi * halfey * dt;
    integralFBz += twoKi * halfez * dt;
    gx += integralFBx;
    gy += integralFBy;
    gz += integralFBz;
}
gx += twoKp * halfex;  // 比例项（快速收敛）
gy += twoKp * halfey;
gz += twoKp * halfez;

// 步骤5: 四元数微分方程积分
gx *= 0.5f * dt;
gy *= 0.5f * dt;
gz *= 0.5f * dt;
qw += (-qb*gx - qc*gy - qz*gz);
qx += ( qa*gx + qc*gz - qz*gy);
qy += ( qa*gy - qb*gz + qz*gx);
qz += ( qa*gz + qb*gy - qc*gx);

// 步骤6: 四元数归一化
recipNorm = invSqrt(qw*qw + qx*qx + qy*qy + qz*qz);
qw *= recipNorm; qx *= recipNorm; qy *= recipNorm; qz *= recipNorm;
```

**关键参数**:
- `twoKp = 60.0` (比例增益, Kp=30, 快速响应)
- `twoKi = 0.2` (积分增益, Ki=0.1, 零偏校正)

**四元数→欧拉角转换**:

```c
void sensfusion6GetEulerRPY(float *roll, float *pitch, float *yaw) {
    // 先计算估计的重力方向
    gravX = 2*(qx*qz - qw*qy);
    gravY = 2*(qw*qx + qy*qz);
    gravZ = qw*qw - qx*qx - qy*qy + qz*qz;
    
    *yaw   = atan2f(2*(qw*qz + qx*qy), qw*qw + qx*qx - qy*qy - qz*qz) * 180/π;
    *pitch = asinf(gravX) * 180/π;     // 鼻子上仰→pitch为正
    *roll  = atan2f(gravY, gravZ) * 180/π; // 右倾→roll为正
}
```

### 6.2 Madgwick 梯度下降算法（可选）

通过定义 `MADWICK_QUATERNION_IMU` 启用。使用梯度下降法最小化加速度计测量值与重力方向估计之间的误差。

### 6.3 快速逆平方根

两种算法都使用了著名的 **Quake III 快速逆平方根**算法：

```c
float invSqrt(float x) {
    float halfx = 0.5f * x;
    float y = x;
    long i = *(long*)&y;
    i = 0x5f3759df - (i >> 1);   // 魔数近似
    y = *(float*)&i;
    y = y * (1.5f - (halfx * y * y));  // 一次牛顿迭代
    return y;
}
```

## 7. 卡尔曼滤波状态估计器

### 7.1 算法概述

系统实现了一个 **扩展卡尔曼滤波器 (EKF)**，基于论文：

> Mueller, Hamer, D'Andrea. "Fusing ultra-wideband range measurements with accelerometers and rate gyroscopes for quadrocopter state estimation" (ICRA 2015)

### 7.2 状态向量

9维状态向量 $\mathbf{S}$：

$$\mathbf{S} = \begin{bmatrix} x \\ y \\ z \\ \dot{x} \\ \dot{y} \\ \dot{z} \\ d_0 \\ d_1 \\ d_2 \end{bmatrix}$$

| 索引 | 符号 | 含义 |
|------|------|------|
| 0-2 | $x, y, z$ | 世界坐标系位置 (m) |
| 3-5 | $\dot{x}, \dot{y}, \dot{z}$ | 世界坐标系速度 (m/s) |
| 6-8 | $d_0, d_1, d_2$ | 姿态误差（小角度近似）|

姿态使用单独的四元数 $\mathbf{q} = [q_0, q_1, q_2, q_3]$ 表示，与姿态误差状态配合使用。

### 7.3 预测步骤

预测以 **500Hz** 执行（原设计100Hz，已优化）：

$$\mathbf{S}_{k|k-1} = f(\mathbf{S}_{k-1|k-1}, \mathbf{u}_k, \Delta t)$$

```c
// 将加速度计读数从体坐标系转换到世界坐标系
// 使用旋转矩阵 R（由四元数计算得到）
float accWorld_x = R[0][0]*ax + R[0][1]*ay + R[0][2]*az;
float accWorld_y = R[1][0]*ax + R[1][1]*ay + R[1][2]*az;
float accWorld_z = R[2][0]*ax + R[2][1]*ay + R[2][2]*az - GRAVITY;

// 速度积分
S[KC_STATE_PX] += accWorld_x * dt;
S[KC_STATE_PY] += accWorld_y * dt;
S[KC_STATE_PZ] += accWorld_z * dt;

// 位置积分
S[KC_STATE_X] += S[KC_STATE_PX] * dt;
S[KC_STATE_Y] += S[KC_STATE_PY] * dt;
S[KC_STATE_Z] += S[KC_STATE_PZ] * dt;

// 四元数更新（陀螺仪积分）
dq = 0.5 * q ⊗ [0, ωx, ωy, ωz] * dt
q = normalize(q + dq)
```

**过程噪声参数**:

| 参数 | 值 | 含义 |
|------|-----|------|
| `procNoiseAcc_xy` | 2.0 | XY加速度过程噪声（降低积分信任度）|
| `procNoiseAcc_z` | 1.0 | Z加速度过程噪声 |
| `measNoiseGyro_rollpitch` | 0.07 rad/s | 陀螺仪噪声（≈4°/s）|
| `measNoiseGyro_yaw` | 0.1 rad/s | 偏航陀螺仪噪声 |

### 7.4 更新步骤（标量更新）

所有传感器观测使用统一的**标量更新**函数，避免了矩阵求逆：

```c
static void scalarUpdate(kalmanCoreData_t *this, xtensa_matrix_instance_f32 *Hm,
                         float error, float stdMeasNoise)
{
    // 1. 计算新息协方差: S = H·P·H' + R
    mat_mult(&P, &H', &PH');     // P·H'
    float HPHR = R;
    for (i = 0..N) HPHR += H[i] * PH'[i];  // H·P·H' + R

    // 2. 计算卡尔曼增益: K = P·H' / S
    for (i = 0..N) K[i] = PH'[i] / HPHR;

    // 3. 状态更新: S = S + K·error
    for (i = 0..N) S[i] += K[i] * error;

    // 4. 协方差更新 (Joseph form): P = (I-K·H)·P·(I-K·H)' + K·R·K'
    // 保证数值对称性和正定性
}
```

## 8. PID 控制系统

### 8.1 控制架构

系统采用**双环级联PID**控制架构：

```
                  外环 (姿态角)            内环 (角速率)
Setpoint ──→ [Attitude PID] ──→ rateDesired ──→ [Rate PID] ──→ Actuator
              250Hz                              250Hz
  ↑                                                ↓
  │          roll_actual                      gyro.x/y/z
  │              ↑                                ↑
  └──── state.attitude ◄──── 估计器 ◄──── 传感器数据
```

### 8.2 PID 算法实现

```c
float pidUpdate(PidObject* pid, const float measured, const bool updateError)
{
    float output = 0.0f;

    // P项: 比例控制
    if (updateError) pid->error = pid->desired - measured;
    pid->outP = pid->kp * pid->error;
    output += pid->outP;

    // D项: 微分控制（带可选低通滤波）
    float deriv = (pid->error - pid->prevError) / pid->dt;
    if (pid->enableDFilter) {
        pid->deriv = lpf2pApply(&pid->dFilter, deriv);  // 二阶LPF去噪
    } else {
        pid->deriv = deriv;
    }
    pid->outD = pid->kd * pid->deriv;
    output += pid->outD;

    // I项: 积分控制（带限幅防饱和）
    pid->integ += pid->error * pid->dt;
    if (pid->iLimit != 0) {
        pid->integ = constrain(pid->integ, -pid->iLimit, pid->iLimit);
    }
    pid->outI = pid->ki * pid->integ;
    output += pid->outI;

    // 总输出限幅
    if (pid->outputLimit != 0) {
        output = constrain(output, -pid->outputLimit, pid->outputLimit);
    }

    pid->prevError = pid->error;
    return output;
}
```

### 8.3 PID 参数配置

**内环 (Rate PID) - 角速率控制**:

| 轴 | Kp | Ki | Kd | iLimit |
|----|----|----|----|----|
| Roll Rate | 130.0 | 0.0 | 5.0 | 33.3 |
| Pitch Rate | 130.0 | 0.0 | 5.0 | 33.3 |
| Yaw Rate | 100.0 | 0.0 | 3.0 | 166.7 |

> **注意**: 内环Ki=0是首飞调参策略，消除桌面测试时的积分饱和问题。飞行稳定后再逐步加入Ki。

**外环 (Attitude PID) - 姿态角控制**:

当前外环PID增益全部设为0，表示系统仍处于**调试阶段**，通过参数系统可在线调整。

### 8.4 controllerPid() 完整控制流程

```c
void controllerPid(control_t *control, setpoint_t *setpoint,
                   const sensorData_t *sensors, const state_t *state,
                   const uint32_t tick)
{
    // ① YAW模式处理
    if (setpoint->mode.yaw == modeVelocity) {
        attitudeDesired.yaw += setpoint->attitudeRate.yaw * dt;
    }

    // ② 位置控制器（如启用）
    positionController(&actuatorThrust, &attitudeDesired, setpoint, state);

    // ③ 外环: 姿态角PID → 期望角速率
    attitudeControllerCorrectAttitudePID(
        state->attitude.roll/pitch/yaw,
        attitudeDesired.roll/pitch/yaw,
        &rateDesired.roll/pitch/yaw);

    // ④ 外环输出限幅 (±100°/s)
    rateDesired.roll  = clamp(rateDesired.roll,  -100, 100);
    rateDesired.pitch = clamp(rateDesired.pitch, -100, 100);

    // ⑤ 内环: 角速率PID → 执行器输出
    attitudeControllerCorrectRatePID(
        sensors->gyro.x/y/z,       // 实际角速率
        rateDesired.roll/pitch/yaw); // 期望角速率

    // ⑥ 获取控制输出
    attitudeControllerGetActuatorOutput(&control->roll, &control->pitch, &control->yaw);

    // ⑦ Yaw符号翻转 (X构型需要)
    control->yaw = -control->yaw;

    // ⑧ 推力比例限幅 (保证电机输出≥0)
    // |R| ≤ 0.6T, |P| ≤ 0.6T, |Y| ≤ 0.2T
    // → 最差电机 ≥ T - 0.3T - 0.3T - 0.2T = 0.2T (至少20%推力)

    // ⑨ 安全联锁: 所有PID增益为0时禁止输出推力
    if (allPidGainsAreZero()) {
        control->thrust = 0;  // 防止无姿态稳定的开环油门
    }

    // ⑩ 油门死区: thrust < 3000 → 清零所有输出 + 重置PID
    if (control->thrust < THRUST_DEAD_ZONE) {
        control->thrust = control->roll = control->pitch = control->yaw = 0;
        attitudeControllerResetAllPID();
    }
}
```

### 8.5 符号链分析

控制符号链确保负反馈正确性：

```
Roll:  飞机右倾(roll>0) → PID输出负R → r<0
       → 右侧(M1,M4)加速 → 右侧上抬 → roll减小 → 正确负反馈 ✓

Pitch: 鼻子下压(pitch<0) → PID输出正P → p>0
       → 前方(M1,M2)加速 → 鼻子上仰 → pitch增大 → 正确负反馈 ✓

Yaw:   X构型CW电机系数与+构型相反
       → 需要对yaw输出取反 → control->yaw = -control->yaw ✓
```

### 8.6 位置控制器（定高 / 定点模式）

#### 8.6.1 功能概述

系统在双环级联姿态控制的外部，额外实现了一层**位置/速度串级 PID 控制器**（`position_controller_pid.c`），支持两种高级飞行模式：

| 飞行模式 | 控制轴 | 必需传感器 | Setpoint 标志 |
|--------|--------|----------|--------------|
| 姿态自稳 | Roll / Pitch / Yaw + 油门 | IMU | `mode.x/y/z = modeDisable` |
| **定高模式** | Z 轴高度锁定，水平手动 | 气压计 (MS5611) / ToF (MTF01) | `mode.z = modeAbs` |
| **定点模式** | XYZ 全锁定，自动悬停 | 光流 + 高度传感器 (MTF01) | `mode.x/y/z = modeAbs` |

飞行模式通过 SBUS CH6 三段开关切换（低位=姿态自稳，中位=定高，高位=定点），也可由远程服务器通过 `CTRL_CMD_HOVER` 指令触发。

#### 8.6.2 控制架构

位置控制器在主 PID 循环开头（`250Hz`）、外环姿态 PID 之前调用：

```c
void controllerPid(...) {
    // ① 位置控制器（当 setpoint->mode.z/x/y != modeDisable 时激活）
    positionController(&actuatorThrust, &attitudeDesired, setpoint, state);

    // ② 外环姿态 PID: 期望角度 → 期望角速率
    attitudeControllerCorrectAttitudePID(...);

    // ③ 内环角速率 PID: 期望角速率 → 执行器输出
    attitudeControllerCorrectRatePID(...);
}
```

位置控制器本质上是**在姿态双环之外再套一层串级**，其输出为期望 Roll/Pitch 角度和推力，替代遥控器的直接输入。

#### 8.6.3 串级 PID 结构

**定点模式（4 环级联）**:

```
位置误差(m) → [pidX/Y] → 期望速度(m/s) → [pidVX/VY] → 期望Roll/Pitch(°)
                                                               ↓
高度误差(m) → [pidZ]   → 期望垂直速度  → [pidVZ]   → 推力补偿
                                                               ↓
                                              [外环attPID] → 期望角速率
                                                               ↓
                                              [内环ratePID] → 电机输出
```

**定高模式（2 环级联，仅 Z 轴）**:

```
高度误差(m) → [pidZ] → 期望垂直速度 → [pidVZ] → 推力
水平方向：遥控器直接给定 Roll/Pitch 指令（位置控制器跳过 XY 轴）
```

#### 8.6.4 控制参数

| PID 环节 | Kp | Ki | Kd | 输出限制 |
|---------|----|----|-----|--------|
| X / Y 位置 | 1.9 | 0.1 | 0 | `xyVelMax × 1.1 = 1.1 m/s` |
| Z 高度 | 1.6 | 0.5 | 0 | `max(zVelMax, 0.5) × 1.1` |
| Vx / Vy 速度 | 25.0 | 1.0 | 0 | `rpLimit × 1.1 = 22°` |
| Vz 垂直速度 | 22.0 | 15.0 | 0 | `UINT16_MAX/2 / thrustScale` |

| 参数 | 值 | 说明 |
|------|-----|------|
| `rpLimit` | 20° | 位置控制器最大 Roll/Pitch 输出 |
| `xyVelMax` | 1.0 m/s | 水平最大速度限制 |
| `zVelMax` | 1.0 m/s | 垂直最大速度限制 |
| `thrustBase` | 24000（无刷）/ 36000（有刷）| 悬停基础推力 Feed-forward |
| `thrustScale` | 1000.0 | 推力比例因子（速度PID输出 → 推力增量）|

#### 8.6.5 传感器依赖

**定高模式**：依赖气压计（MS5611）或 ToF 测距（MTF01）提供 Z 轴观测，卡尔曼滤波器融合后输出 `state.position.z` 和 `state.velocity.z`。气压计启动时需 50 个样本（约 2 秒）完成基准校准，校准后精度约 ±0.5 m；ToF 精度可达 ±5 cm，室内飞行优先使用。

**定点模式**：额外依赖光流传感器（MTF01）提供 XY 速度观测：
- MTF01 测量机体坐标系速度 Vx/Vy（cm/s），经高度补偿后转换至世界坐标系
- 通过 `kalmanCoreUpdateWithVelocity()` 注入 EKF，标准差 `flowStdDev = 0.05 m/s`
- ToF 测距同时用于光流比例补偿（视觉速度 ∝ 飞行高度）

> ⚠️ 若无传感器支持而强制切换至定高/定点模式，位置 PID 将因状态估计不可靠而发散，须确认传感器工作正常。

#### 8.6.6 远程服务器悬停指令

除 SBUS 拨杆切换外，远程服务器可直接下发 `CTRL_CMD_HOVER` 触发悬停模式：

```c
case CTRL_CMD_HOVER:
    setpoint.mode.x = modeVelocity;   // 水平速度锁零
    setpoint.mode.y = modeVelocity;
    setpoint.mode.z = modeAbs;         // 高度绝对位置锁定
    setpoint.velocity.x = 0;
    setpoint.velocity.y = 0;
    float targetHeight = cmd->thrust / 1000.0f;
    targetHeight = clamp(targetHeight, 0.1f, 3.0f);  // 安全范围 0.1~3.0 m
    setpoint.position.z = targetHeight;
    commanderSetSetpoint(&setpoint, COMMANDER_PRIORITY_REMOTE);
    break;
```

## 9. 电机混控与功率分配

### 9.1 电机驱动

使用ESP32的 **MCPWM** (Motor Control PWM) 外设驱动无刷ESC：

```c
// ESC参数
#define ESC_TIMEBASE_PERIOD   2500    // 2.5ms → 400Hz
#define ESC_PULSE_MIN_US      1000    // 最小脉宽 (0%油门)
#define ESC_PULSE_MAX_US      2000    // 最大脉宽 (100%油门)

// 推力值(0~65535) → 脉宽(1000~2000μs) 线性映射
static uint32_t thrustToPulseWidth(uint16_t thrust) {
    return 1000 + (thrust * 1000) / 65535;
}
```

每2个电机共用一个MCPWM组（ESP32-S3有2个MCPWM组），4个电机需要2个组。

### 9.2 X构型混控公式

$$M_1 = T - \frac{R}{2} + \frac{P}{2} - Y \quad \text{(前右, CW)}$$
$$M_2 = T + \frac{R}{2} + \frac{P}{2} + Y \quad \text{(前左, CCW)}$$
$$M_3 = T + \frac{R}{2} - \frac{P}{2} - Y \quad \text{(后左, CW)}$$
$$M_4 = T - \frac{R}{2} - \frac{P}{2} + Y \quad \text{(后右, CCW)}$$

其中 $T$ = thrust, $R$ = roll, $P$ = pitch, $Y$ = yaw。

### 9.3 输出保护

```c
// 当修正量总和超过推力时，等比例缩放
int32_t totalCorrection = |r| + |p| + |y|;
if (thrust > 0 && totalCorrection > thrust) {
    float scale = (float)thrust / (float)totalCorrection;
    r = r * scale;
    p = p * scale;
    y = y * scale;
}
```

这确保了所有电机输出值非负，防止低油门时PID输出导致的异常。

## 10. 遥控输入 (SBUS)

### 10.1 SBUS协议

SBUS使用反相UART信号，100000波特率，8E2格式：

| 字段 | 长度 | 说明 |
|------|------|------|
| Header | 1字节 | 0x0F |
| Channels | 22字节 | 16个11位通道数据（位压缩） |
| Flags | 1字节 | failsafe/frameLost标志 |
| Footer | 1字节 | 0x00 |

**通道解析**:

```c
frame->channels[0]  = ((buffer[1]       | buffer[2]<<8)  & 0x07FF);
frame->channels[1]  = ((buffer[2]>>3    | buffer[3]<<5)  & 0x07FF);
frame->channels[2]  = ((buffer[3]>>6    | buffer[4]<<2   | buffer[5]<<10) & 0x07FF);
frame->channels[3]  = ((buffer[5]>>1    | buffer[6]<<7)  & 0x07FF);
// ... 以此类推，11位一组密集打包
```

### 10.2 通道映射与控制转换

```c
// SBUS通道值范围: 172 ~ 1811 (11-bit, 中心 ≈ 992)
#define EXTRX_SIGN_ROLL   (-1)
#define EXTRX_SIGN_PITCH  (-1)
#define EXTRX_SIGN_YAW    (-1)
#define EXTRX_SCALE_ROLL   20.0f   // 度

#define EXTRX_SCALE_YAW    150.0f  // 度/秒

}

// 推力通道: SBUS值 → 0~65535

}
```

## 11. 通信系统

### 11.1 CRTP (Crazy RealTime Protocol)

CRTP是Crazyflie的核心通信协议，通过Wi-Fi UDP传输：

```c
// CRTP包结构
typedef struct {
    uint8_t size;
    union {
        struct {
            uint8_t header;    // [port:4][channel:2][reserved:2]
            uint8_t data[30];
        };
    };
} CRTPPacket;
```

支持的端口:
- Port 0x02: Commander (遥控指令)
- Port 0x05: Log (日志数据)
- Port 0x06: Param (参数读写)
- Port 0x07: Mem (内存操作)

### 11.2 Wi-Fi 通信

ESP32同时支持AP和STA模式：
- **AP模式**: 创建热点，地面站直接连接
- **STA模式**: 连接到现有Wi-Fi网络，支持远程服务器通信

### 11.3 远程服务器

远程服务器模块（`remote_server.c`）是该项目面向地面站/云端的一条完整数据链路。它不是简单的 socket 封装，而是一个集**发现、连接、控制、遥测、故障恢复**于一体的通信子系统。

#### 11.3.1 总体架构

模块采用 **TCP 控制 + UDP 遥测** 的双通道设计：

```
地面站/服务器
   ├─ UDP 8081 广播发现响应
   ├─ TCP 主连接：控制命令 / CRTP / 心跳 / 配置
   └─ UDP 遥测端口：高速状态上传

ESP32 飞控
   ├─ remoteServerTxTask()         发送任务
   ├─ remoteServerRxTask()         接收解析任务
   └─ remoteServerTelemetryTask()  周期遥测任务
```

#### 11.3.2 自动发现机制

模块上电后优先通过 UDP 广播在局域网内寻找服务器，而不是直接依赖硬编码 IP：

- 发现端口：`8081`
- 魔数：`0x45535044`，即 ASCII `"ESPD"`
- 超时时间：`5000 ms`
- 成功后记录服务器 IP 与 UDP 遥测端口

设计意义：
- 适合移动热点、路由器 DHCP 等动态地址环境
- 减少固件重烧录或手工改 IP 的频率
- 若连接断开，会清除发现状态，下一次重连重新广播寻找服务器

这也是它区别于普通 Wi-Fi 透传模块的重要特征：**服务端地址不是静态假设，而是运行时协商得到的。**

#### 11.3.3 连接状态机与重连策略

内部维护连接状态：

```
REMOTE_STATE_DISCONNECTED
    ↓
REMOTE_STATE_CONNECTING
    ↓
REMOTE_STATE_CONNECTED
    ↓
   （错误/超时/对端关闭）
    ↓
REMOTE_STATE_DISCONNECTED
```

关键机制：
- `connectToServer()` 负责发现、建 TCP socket、设置收发超时并发起连接
- `disconnectFromServer()` 同时关闭 TCP/UDP socket，并重置发现结果
- `eventGroup` 使用 `EVT_CONNECTED / EVT_DISCONNECTED / EVT_RECONNECT` 协调三条任务
- TX 任务在连接失败时按 `CONFIG_REMOTE_SERVER_RECONNECT_INTERVAL` 周期自动重试
- 心跳周期由 `CONFIG_REMOTE_SERVER_HEARTBEAT_INTERVAL` 控制

这种设计避免了在飞控主线程里直接处理网络异常，把抖动和重连成本隔离在后台任务中。

#### 11.3.4 三任务并发模型

| 任务 | 优先级 | 栈 | 作用 |
|------|--------|----|------|
| `remoteServerTxTask` | 3 | 4096 | 建链、发送队列数据、发送心跳 |
| `remoteServerRxTask` | 3 | 4096 | 接收 TCP 数据流、按包头重组并分发 |
| `remoteServerTelemetryTask` | 2 | 3072 | 周期采集日志变量并通过 UDP 上传 |

这种拆分有三个好处：
- **发送与接收解耦**：避免阻塞型 socket 调用互相影响
- **遥测与控制分离**：高频状态上传不会挤占控制命令带宽
- **更容易定位故障**：链路问题可从 TX/RX/Telemetry 三个方向分别排查

#### 11.3.5 TCP 主协议封装

TCP 通道上的所有数据均带统一包头：

```c
typedef struct {
    uint8_t  magic[2];
    uint8_t  version;
    uint8_t  type;
    uint16_t seq;
    uint16_t length;
} RemotePacketHeader;
```

典型包类型包括：
- `REMOTE_PKT_HEARTBEAT`：心跳/保活
- `REMOTE_PKT_CONTROL`：遥控与模式指令
- `REMOTE_PKT_CRTP`：Crazyflie CRTP 数据透传
- `REMOTE_PKT_ACK`：确认包
- `REMOTE_PKT_CONFIG`：运行时配置命令

发送路径中，`sendPacketInternal()` 会先发送包头，再发送 payload，并使用 `sockMutex` 保护 TCP socket，避免多个任务并发写同一连接。

#### 11.3.6 接收解析与流重组

TCP 是字节流而非报文，因此接收任务需要自己做“拆包/粘包处理”：

- 使用 `rxBuffer[256]` 持续接收
- 若包头魔数不匹配，则逐字节滑动重新同步
- 当缓冲区内数据不足一个完整包时，保留残留数据等待下次接收
- 收到完整包后调用 `processReceivedPacket()` 分发

这是一个标准但非常关键的网络工程细节。若省略这层流重组，控制包在高吞吐情况下会极易错位或误解析。

#### 11.3.7 控制命令与安全约束

远程服务器并不是直接把网络数据“原样送进控制器”，而是经过完整的安全校验后再生成 `setpoint_t`：

| 限幅项 | 限值 |
|------|------|
| Roll | ±45° |
| Pitch | ±45° |
| Yaw Rate | ±200°/s |
| Thrust | 0 ~ 60000 |

`handleControlPacket()` 支持的主要指令：
- `CTRL_CMD_RPYT`：姿态+偏航角速率+推力控制
- `CTRL_CMD_HOVER`：进入悬停/定高模式
- `CTRL_CMD_LAND`：触发下降着陆（`vz = -0.3 m/s`）
- `CTRL_CMD_ARM / DISARM`：解锁 / 上锁
- `CTRL_CMD_EMERGENCY`：紧急停机
- `CTRL_CMD_SET_CONTROL_MODE`：切换远程控制模式

其中 `validateControlCmd()` 先做数据合法性检查，若超范围或疑似数据损坏，命令会被直接丢弃并累计到 `invalidCmdCount`。

#### 11.3.8 与 Commander / VehicleState 的耦合方式

远程命令最终不是直接改电机，而是走与本地控制一致的上层接口：

- 正常遥控指令通过 `commanderSetSetpoint()` 注入 Commander
- 使用 `COMMANDER_PRIORITY_REMOTE` 参与多源仲裁
- 解锁/上锁/紧急停机通过 `vehicleArm()`、`vehicleDisarm()`、`vehicleEmergencyStop()` 进入统一状态机
- 网络连接状态还会同步到 `vehicleSetGcsConnected()`，作为飞行状态的一部分对外暴露

这意味着远程服务器是**系统架构中的一等控制源**，而不是旁路控制通道；它遵守同样的优先级和安全链路。

#### 11.3.9 UDP 高频遥测链路

为避免大量状态上传占满 TCP 带宽，遥测走单独 UDP socket：

- UDP 包头魔数：`ESPU`
- 包头长度：6 字节
- 内容格式：`[magic(4)][type(1)][seq(1)][payload]`
- 发送函数：`sendUDPTelemetry()`

遥测任务会从日志系统批量抓取状态量，组装 `RemoteTelemetryData` 后周期发送。采集内容包括：

- 姿态角：Roll / Pitch / Yaw
- 陀螺仪与加速度计原始量
- 电机输出功率 `M1~M4`
- 飞行状态机：arming / flightMode / flightPhase / failsafe
- 控制来源：遥控器、远程地面站或无控制
- 高度与速度：`stateEstimate.z/vx/vy/vz`
- 气压高度、ToF 距离、机体坐标系速度

这种设计把“可靠控制”和“高频观测”分层：
- **TCP** 负责必须到达的数据
- **UDP** 负责允许偶发丢包但要求低延迟的数据

非常符合飞控遥测系统的工程需求。

#### 11.3.10 CRTP 远程透传

除了自定义控制命令外，远程服务器还支持 `REMOTE_PKT_CRTP`，把网络侧收到的 CRTP 包重新注入本地 Crazyflie 协议栈：

- 服务端可远程访问 `Commander / Log / Param / Mem` 等 CRTP 端口
- ESP32 端通过 `crtpInjectPacket()` 注入现有 CRTP 处理链
- 等效于把 Crazyflie 上位机生态“桥接”到 Wi-Fi/服务器侧

这使项目兼容传统 Crazyflie 工具链，也为后续参数调试、日志订阅和远程配置提供了扩展空间。

#### 11.3.11 控制模式与多源协同

模块内部维护 `remoteControlMode`，支持：

- `REMOTE_CTRL_MODE_ENABLED`：无本地遥控时允许远程接管
- `REMOTE_CTRL_MODE_SHARED`：与 SBUS 遥控协同工作

遥测中还会根据 `commanderGetActivePriority()`、遥控连接状态、远程连接状态，推断当前实际控制来源并回传给地面站。这个细节对“谁在控制飞机”的诊断非常重要。

#### 11.3.12 统计与运维信息

模块维护多项链路统计量：

- `txPacketCount`
- `rxPacketCount`
- `reconnectCount`
- `invalidCmdCount`
- `connectionStartTime`

这些指标可用于地面站显示链路质量、分析断连频率、定位控制命令异常来源。

### 12.2 解锁检查

```c
#define ARM_CHECK_ENABLED (ARM_CHECK_SENSORS | ARM_CHECK_THROTTLE_ZERO)

static bool runArmingChecks(bool force) {
    // 1. 传感器健康检查
    if (flags & ARM_CHECK_SENSORS) {
        if (!sensorsAreCalibrated()) fail("Sensors not ready");
    }
    // 2. 电池电压检查 (可选)
    if (flags & ARM_CHECK_BATTERY) {
        if (voltage < 3300mV) fail("Battery low");
    }
    // 3. 油门归零检查
    if (flags & ARM_CHECK_THROTTLE_ZERO) {
        if (throttle > threshold) fail("Throttle not zero");
    }
}
```

| IN_AIR | 持续飞行中 |
| LANDING | 油门降低 |
| LANDED | 速度<0.1m/s 持续1秒 |

### 12.4 飞行模式/控制层级对照表

当前工程里，“遥控模式”“`setpoint.mode` 标志”“`vehicleState.flightMode` 状态”和“实际被激活的控制器层级”并不完全等价。为了避免阅读源码时混淆，可按下表理解：

| 输入来源 | 典型触发 | `setpoint.mode` | `vehicleState.flightMode` | 激活控制层 |
|---------|----------|-----------------|---------------------------|-----------|
| SBUS 手动/姿态 | CH6 低位或普通遥杆 | `x/y/z=modeDisable` | `STABILIZE` | 姿态外环 + 角速率内环 |
| SBUS 定高 | CH6 中位，具备高度观测 | `z=modeAbs`, `x/y=modeDisable` | `ALTITUDE` | 高度位置环 + 垂直速度环 + 姿态双环 |
| SBUS 定点 | CH6 高位，具备光流/ToF | `x/y/z=modeAbs` | `POSITION` | 位置环 + 速度环 + 姿态双环 |
| 远程悬停 | `CTRL_CMD_HOVER` | `x/y=modeVelocity(0)`, `z=modeAbs` | 通常表现为 `ALTITUDE/POSITION` 语义 | 水平零速度 + 高度锁定 |
| 远程降落 | `CTRL_CMD_LAND` | `z=modeVelocity`, `vz=-0.3` | `LAND` | 垂直速度控制 |
| 紧急停机 | `CTRL_CMD_EMERGENCY` 或软件触发 | 所有轴禁用，推力清零 | `STANDBY_ERROR` / Emergency | 控制链立即切断 |

需要注意：
- `vehicleSetFlightMode()` 负责把高层飞行模式映射到旧的 `setCommandermode()` 体系；
- 不是所有枚举模式都已经实现完整任务逻辑，例如 `MISSION`、`RTL`、`OFFBOARD` 目前更多是状态接口预留；
- 真正是否进入位置闭环，最终仍由 `setpoint->mode.x/y/z` 是否为 `modeAbs` 决定。

## 13. 安全保护机制

### 13.1 多层安全体系

| 层级 | 机制 | 触发条件 | 动作 |
|------|------|---------|------|
| L1 | 硬件看门狗 | 稳定器3秒无响应 | 系统重启 |
| L2 | 紧急停机 | 软件触发 | 电机立即停止 |
| L3 | PID零增益联锁 | 所有PID增益=0 | 禁止推力输出 |
| L4 | 油门死区 | thrust < 3000 | 清零输出+重置PID |
| L5 | 控制输出限幅 | R/P/Y超限 | 等比例缩放 |
| L6 | 电机输出保护 | 混控后电机<0 | 钳位到0 |
| L7 | 信号丢失检测 | SBUS帧丢失>80% | 切换到安全模式 |
| L8 | 指令超时 | 1秒无指令 | 控制源故障转移 |

### 13.2 Commander 优先级仲裁

```c
// 优先级: CRTP < HIGH_LEVEL < EXTRX(SBUS)
// 带锁定机制防止抖动
// 支持控制源健康监控和自动故障转移

void commanderSetSetpoint(setpoint_t *setpoint, int priority) {
    if (priority >= currentPriority) {
        // 验证setpoint合法性（NaN/Inf/超限检查）
        commanderValidateSetpoint(setpoint);
        // 钳位到安全范围
        commanderClampSetpoint(setpoint);
        // 接受设定值
        xQueueOverwrite(setpointQueue, setpoint);
    }
}
```

**安全限制**:
- 最大Roll/Pitch: ±45°
- 最大Yaw速率: ±200°/s  
- 最大推力: 60000 (约91.5%)

### 13.3 失效降级路径

项目当前已经具备“检测异常 → 切换控制源/模式 → 保持安全输出”的基础框架，但不同故障的处理深度并不完全一致。

| 故障场景 | 检测来源 | 当前动作 | 降级结果 |
|---------|---------|---------|---------|
| SBUS 帧丢失/失败保护 | SBUS flags + 滑窗统计 | `vehicleSetRcConnected(false)`，`failsafe=RC_LOSS` | 优先切换到远程控制；若无其他控制源，仅保留安全状态 |
| 当前控制源 1 秒无更新 | `commander.c` 健康监控 | `performSourceFailover()` 重新选择可用优先级 | `EXTRX → REMOTE → CRTP → DISABLE` |
| 远程 TCP 断链 | `recv/send` 错误、心跳失败 | 置 `EVT_DISCONNECTED`，后台自动重连 | 地面站链路中断，但飞控主循环持续运行 |
| 遥测 UDP 丢包 | UDP 无连接发送 | 不重发、仅丢样本 | 只影响监控连续性，不影响控制闭环 |
| ToF 无效/超距 | `tofStatus`/量程检查 | 停止向 EKF 注入该次高度观测 | 高度估计退回气压计或已有状态传播 |
| 光流质量差/无效 | `flowStatus`/`flowQuality` | 不提交 `velocityMeasurement_t` | 定点能力下降，XY 漂移增大 |
| 传感器未校准 | `sensorsAreCalibrated()` | 禁止解锁 | 停留在 `INIT/STANDBY` |
| 紧急停机 | 软件命令 | `stabilizerSetEmergencyStop()` + `systemSetArmed(false)` | 电机立即停止 |

当前设计的关键特点是：
- **控制源故障**优先走 Commander 的仲裁/切换机制；
- **观测源故障**优先表现为 EKF 更新项减少，而不是直接停机；
- **不可接受风险**（例如紧急停机）才直接切断推力。

这说明本项目已经具备基础的分层容错思想：并非所有异常都一律停机，而是按“通信层 / 观测层 / 执行层”分别处理。

不过也要看到局限：
- RC 丢失后尚未实现自动降落策略；
- 电池 failsafe 逻辑目前被显式弱化，因为项目尚未接入可靠电压测量；
- 光流/ToF 丢失后没有自动回退到“姿态模式”的完整闭环策略，更多依赖操作者及时接管。

### 14.4 实时性与性能预算

从时序上看，当前系统的关键控制链大致如下：

| 链路环节 | 频率 | 说明 |
|---------|------|------|
| BMI088 中断 / 采样 | 1 kHz | 由 IMU 数据就绪信号驱动 |
| Sensors 任务 | 1 kHz | SPI 读取 + 坐标对齐 + 基础滤波 |
| Stabilizer 主循环 | 1 kHz 调度 | 系统主时基 |
| EKF 预测 | 500 Hz | 减少积分漂移 |
| 姿态/角速率 PID | 500 Hz | `ATTITUDE_RATE = RATE_500_HZ` |
| 位置控制器 | 100 Hz | `POSITION_RATE = RATE_100_HZ` |
| MTF01 光流/ToF | 100 Hz | 与位置控制速率匹配 |
| ESC PWM | 400 Hz | 1000~2000 μs 脉宽输出 |

一个近似的端到端控制延迟链可理解为：

```text
IMU采样(1ms内) → Sensors任务 → EKF/姿态估计(2ms级) → PID计算(2ms级) → Mixer → PWM更新(2.5ms周期)
```

因此，从姿态扰动发生到 PWM 反映到电机，主控制链路的典型延迟量级约为 **数毫秒到 10 毫秒以内**。对于小型室内四旋翼，这是合理且可飞的时延预算。

双核分工的意义在于：
- **Core 1** 放置 1kHz 级实时任务，保证控制确定性；
- **Core 0** 处理 EKF、SBUS、Wi-Fi、Remote Server、日志等次实时任务；
- 即使网络任务出现抖动，也不直接阻塞稳定器主循环。

报告中最值得强调的一点是：本项目的实时性不是单靠“CPU 主频高”获得的，而是依赖**任务优先级、核间分工、队列隔离和限频执行**共同实现。

## 15. 参数系统与日志观测

### 15.1 参数系统（Param）

项目继承了 Crazyflie 风格的参数系统。各模块通过 `PARAM_GROUP_START()` / `PARAM_ADD()` 把内部变量导出为运行时可读写参数，无需重新编译即可在线调节。

典型参数组包括：

| 参数组 | 主要内容 | 作用 |
|-------|---------|------|
| `pid_attitude` | Roll/Pitch/Yaw 外环 PID | 姿态角控制调参 |
| `pid_rate` | Roll/Pitch/Yaw 内环 PID | 角速度闭环调参 |
| `posCtlPid` | X/Y/Z 位置环、`rpLimit`、`xyVelMax` | 定高/定点调参 |
| `velCtlPid` | Vx/Vy/Vz 速度环 | 悬停与速度闭环调参 |
| `controller` | `tiltComp` 等控制选项 | 控制补偿开关 |
| `commander` | `enHighLevel` | 高层命令功能开关 |
| `vehicle` | 飞行模式/解锁状态只读或半只读映射 | 状态可视化 |

参数系统的工程价值在于：
- 把“算法实现”和“调参过程”解耦；
- 允许首飞阶段快速修正 PID，而不改源码；
- 地面站、CRTP 和远程服务器都可复用这套接口。

### 15.2 日志系统（Log）

日志系统通过 `LOG_GROUP_START()` / `LOG_ADD()` 导出内部状态，用于上位机订阅、调参和飞行回放分析。

当前项目中较关键的日志组包括：

| 日志组 | 典型变量 | 用途 |
|-------|----------|------|
| `controller` | `cmd_thrust`、`rollRate`、`actuatorThrust` | 看控制器输出是否饱和 |
| `pid_attitude` / `pid_rate` | `outP/outI/outD` | 区分 P/I/D 谁在主导振荡 |
| `posCtl` | `targetX/Y/Z`、`Xp/Yp/Zp`、`VXp/VZp` | 定高/定点调参核心观测 |
| `vehicle` | `armState`、`flightMode`、`failsafe` | 看飞行状态切换 |
| `commander` | `activePri`、`accepted/rejected`、`remoteAlive` | 看控制源仲裁与故障转移 |
| `kalman` / `stateEstimate` | 位置、速度、高度 | 看估计器是否稳定 |

远程服务器的 UDP 遥测，本质上就是对日志系统的再封装：它把姿态、IMU、电机、高度、状态机等关键日志变量打包发给地面站。

### 15.3 推荐观测量

如果目标是快速判断系统问题，建议优先看以下变量：

| 问题类型 | 首选观测量 |
|---------|-----------|
| 姿态振荡 | `pid_rate.*_outP/outD`、`gyro.x/y/z` |
| 悬停高度上下波动 | `stateEstimate.z`、`posCtl.Zp/Zi`、`mtf01.distance` |
| 定点横向漂移 | `stateEstimate.vx/vy`、`posCtl.targetX/Y`、光流质量 |
| 控制源抢占异常 | `commander.activePri`、`remoteAlive`、`extRxAlive` |
| 无法解锁 | `vehicle.armState`、`vehicle.failsafe`、传感器校准状态 |

一句话概括：**参数系统负责“改”，日志系统负责“看”，二者合在一起才构成完整的飞控调试闭环。**

## 16. 配置、构建与功能裁剪

### 16.1 Kconfig / sdkconfig 的作用

本项目大量功能不是写死在代码里的，而是通过 `Kconfig.projbuild` → `sdkconfig` 生成编译期宏，再由各模块按需启用。

典型开关包括：

| 配置项 | 作用 |
|-------|------|
| `TARGET_ESP32_S2_DRONE_V1_2` 等 | 选择硬件版本和引脚映射 |
| `ENABLE_POSITION_HOLD_MODE` | 编译启用位置保持能力 |
| `WIFI_ENABLE_STA_MODE` | 启用 STA+AP 双模式联网 |
| `REMOTE_SERVER_ENABLE` | 编译启用远程服务器模块 |
| `REMOTE_SERVER_*` | 远程服务器地址、重连、心跳、遥测周期 |
| `CONFIG_ENABLE_MTF01` | 编译启用光流/ToF 任务 |
| `PITCH_CALIB` / `ROLL_CALIB` | 安装误差静态补偿 |

这意味着报告中的很多能力并不是“所有固件实例天然都具备”，而是与当前 `sdkconfig` 构建结果强相关。

### 16.2 功能裁剪思想

项目的软件结构明显支持按能力裁剪：
- 没有 MTF01 时，可退化为姿态 + 气压定高；
- 没有 STA 模式时，可仅保留 AP 本地通信；
- 关闭远程服务器后，飞控主体依然可以独立运行；
- 硬件版本切换主要通过 Kconfig 和驱动层适配完成，而不是复制一套固件。

这种“模块化 + 编译期裁剪”的设计，对嵌入式项目非常重要：既减小资源占用，也降低功能耦合。

### 16.3 构建与调试建议

建议在报告中明确维护流程：
- 使用 ESP-IDF 的 `menuconfig` 管理配置；
- 变更传感器、网络、硬件版本时，优先检查 `sdkconfig` 是否同步；
- 新功能验收前，先确认对应 `CONFIG_*` 宏已打开，否则源码存在但功能可能并未真正编译进固件。

## 17. 调参与验证建议

### 17.1 建议的调参顺序

当前工程最合理的调参顺序不是“一次把所有模式都调好”，而是分层进行：

1. **仅姿态模式首飞**：先确认 IMU 方向、混控符号、Yaw 反向都正确；
2. **内环 Rate PID**：先调到快速但不过振；
3. **外环 Attitude PID**：再让角度跟踪平稳；
4. **定高 Z 轴**：在 ToF/气压稳定后单独调高度；
5. **定点 XY**：最后再调光流相关位置环。

这与项目当前“外环部分参数仍处于调试阶段”的代码现状是吻合的。

### 17.2 典型验证清单

| 验证项 | 通过标准 |
|-------|---------|
| 解锁前静止姿态 | Roll/Pitch 零点稳定，无明显漂移 |
| 空载油门抬升 | 四电机同步起转，无单轴明显偏转 |
| 小角度指令响应 | Roll/Pitch 命令方向正确，松杆能回正 |
| 低空定高 | 高度波动可控，无持续积分上冲 |
| 低空定点 | 横向漂移缓慢可控，无快速发散 |
| 通信切换 | 遥控 / 远程 / 断链切换时不出现危险突变 |

### 17.3 常见异常与排查方向

| 现象 | 优先排查 |
|------|---------|
| 起飞即翻 | 电机编号、桨向、坐标轴、Yaw 符号链 |
| 高频抖动 | Rate P 过高、D 噪声大、IMU 振动 |
| 缓慢漂移 | 陀螺零偏、姿态零点、光流比例或安装角 |
| 定高忽上忽下 | ToF 无效、Z 环积分过大、`thrustBase` 偏差 |
| 定点打圈 | 光流坐标映射错误、Yaw 漂移、XY 速度环过强 |
| 远程控制偶发失效 | `commander.activePri`、控制源超时、TCP 重连 |

报告加入这类“从现象到模块”的排查表，会比单纯列算法更贴近实际维护工作。

## 18. 系统边界与当前限制

为了避免误解项目能力，建议明确写出当前边界：

### 18.1 已具备的能力

- 基于 BMI088 的姿态稳定控制；
- 基于 EKF + ToF/光流的室内定高/定点；
- SBUS、Wi-Fi、远程服务器多控制源接入；
- Commander 优先级仲裁与基本故障转移；
- 参数在线调节与日志/遥测观测。

### 18.2 尚未完整实现或仅部分预留的能力

- `MISSION`、`RTL`、`OFFBOARD` 等模式在状态枚举中存在，但当前并无完整自主任务逻辑；
- 未集成 GPS，因此不具备室外全局导航和返航闭环；
- RC 丢失后尚未实现自动降落或盘旋等待；
- 电池 failsafe 因测量链路不完备而被弱化；
- 光流/ToF 失效后的自动模式回退仍不完善；
- 性能评估更多基于任务频率与结构分析，缺少系统化的 CPU/栈水位统计报告。

### 18.3 适用场景判断

基于当前实现，更适合把本项目定义为：

> **一套面向室内/近距离实验环境的 ESP32 四旋翼飞控固件，重点在姿态稳定、基础悬停、联网调试和算法研究。**

而不是完整意义上的“室外自主导航飞控”。
