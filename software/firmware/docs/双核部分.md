## ESP-Drone 项目双核使用情况

### 结论
**项目配置为双核模式，但实际上没有主动利用双核特性。**

### 配置层面（启用双核）
```
CONFIG_FREERTOS_UNICORE=n          # 未启用单核模式
CONFIG_FREERTOS_NUMBER_OF_CORES=2  # 双核配置
```

### 任务创建方式（未指定核心）
项目中所有任务都使用 `STATIC_MEM_TASK_CREATE` 宏创建：

```c
// static_mem.h 中的定义
#define STATIC_MEM_TASK_CREATE(NAME, FUNCTION, TASK_NAME, PARAMETERS, PRIORITY) \
    xTaskCreateStatic(...)  // 注意：使用的是 xTaskCreateStatic
```

**关键点**：使用的是 `xTaskCreateStatic`，**而不是** `xTaskCreateStaticPinnedToCore`。这意味着：
- 所有任务都使用 `tskNO_AFFINITY`（无核心亲和性）
- FreeRTOS 调度器会自动在两个核心间分配任务

### 核心分配情况

| 任务类型                             | 核心分配 | 说明                                    |
| ------------------------------------ | -------- | --------------------------------------- |
| **WiFi 相关任务**                    | 核心 0   | ESP-IDF WiFi 驱动强制绑定               |
| **ESP-NOW 任务**                     | 核心 1   | 仅 managed_components 中有 PinnedToCore |
| **stabilizer, kalman, commander 等** | 自动分配 | 使用 tskNO_AFFINITY                     |

### 为什么这样设计？

这个项目是从 **Crazyflie** 移植来的，原版 Crazyflie 运行在 **STM32F4**（单核 ARM Cortex-M4）上。移植到 ESP32 时：
- 保留了原有的静态内存分配模式
- 没有针对双核进行优化重构
- 依赖 FreeRTOS 自动负载均衡

### 实际运行效果

虽然没有显式指定核心，但系统**确实在使用双核**：
- **核心 0**：主要运行 WiFi 协议栈、部分飞控任务
- **核心 1**：运行其他飞控任务（调度器自动分配）

### 潜在优化建议

如果想更好地利用双核，可以考虑：

```c
// 将关键任务绑定到核心 1（避免被 WiFi 干扰）
#define STATIC_MEM_TASK_CREATE_PINNED(NAME, FUNCTION, TASK_NAME, PARAMETERS, PRIORITY, CORE) \
    xTaskCreateStaticPinnedToCore((FUNCTION), (TASK_NAME), \
        osSys_ ## NAME ## StackDepth, (PARAMETERS), (PRIORITY), \
        osSys_ ## NAME ## StackBuffer, &osSys_ ## NAME ## TaskBuffer, (CORE))
```

**推荐的核心分配策略**：
- **核心 0**：WiFi、日志、非关键任务
- **核心 1**：stabilizerTask、kalmanTask（高实时性需求）

## xTaskCreateStatic vs xTaskCreateStaticPinnedToCore

### 函数原型对比

```c
// 标准 FreeRTOS（不指定核心）
TaskHandle_t xTaskCreateStatic(
    TaskFunction_t pxTaskCode,
    const char *pcName,
    uint32_t ulStackDepth,
    void *pvParameters,
    UBaseType_t uxPriority,
    StackType_t *puxStackBuffer,
    StaticTask_t *pxTaskBuffer
);

// ESP-IDF 扩展（指定核心）
TaskHandle_t xTaskCreateStaticPinnedToCore(
    TaskFunction_t pxTaskCode,
    const char *pcName,
    uint32_t ulStackDepth,
    void *pvParameters,
    UBaseType_t uxPriority,
    StackType_t *puxStackBuffer,
    StaticTask_t *pxTaskBuffer,
    BaseType_t xCoreID          // ← 多了这个参数
);
```

### 核心区别

| 特性     | xTaskCreateStatic         | xTaskCreateStaticPinnedToCore |
| -------- | ------------------------- | ----------------------------- |
| 核心绑定 | 无（等于 tskNO_AFFINITY） | 可指定 0、1 或 tskNO_AFFINITY |
| 来源     | 标准 FreeRTOS API         | ESP-IDF 扩展 API              |
| 调度     | 任意核心运行              | 绑定到指定核心                |

## tskNO_AFFINITY 的作用

```c
#define tskNO_AFFINITY  ((BaseType_t) 0x7FFFFFFF)
```

**含义**：任务没有核心亲和性，可以在**任意核心**上运行。

```c
// 这两个调用效果相同：
xTaskCreateStatic(...);
xTaskCreateStaticPinnedToCore(..., tskNO_AFFINITY);
```

**调度行为**：
- 调度器根据**负载均衡**自动选择核心
- 任务可能在运行过程中**在两个核心间切换**
- 适合对实时性要求不高的任务

## ESP32 FreeRTOS 双核原理

### 硬件基础

```
┌─────────────────────────────────────────┐
│              ESP32 / ESP32-S3            │
├────────────────────┬────────────────────┤
│     Core 0         │      Core 1        │
│  (PRO_CPU/协议CPU) │  (APP_CPU/应用CPU) │
│  Xtensa LX6/LX7    │  Xtensa LX6/LX7    │
├────────────────────┴────────────────────┤
│           共享 SRAM (520KB)              │
│           共享外设 (SPI, I2C, UART...)   │
└─────────────────────────────────────────┘
```

### ESP-IDF 对 FreeRTOS 的修改

标准 FreeRTOS 是**单核**的，ESP-IDF 进行了深度修改：

```c
// 标准 FreeRTOS: 一个就绪列表
static List_t pxReadyTasksLists[configMAX_PRIORITIES];

// ESP-IDF SMP FreeRTOS: 每个核心一个调度器状态
// 使用自旋锁保护共享资源
```

**关键修改**：

| 组件     | 标准 FreeRTOS | ESP-IDF SMP FreeRTOS   |
| -------- | ------------- | ---------------------- |
| 调度器   | 1个           | 2个（每核一个）        |
| 就绪列表 | 1套           | 共享，带核心亲和性过滤 |
| 中断     | 单核处理      | 双核分配               |
| 锁机制   | 不需要        | 自旋锁 + 临界区保护    |

### 双核调度流程

```
┌─────────────────┐     ┌─────────────────┐
│    Core 0       │     │    Core 1       │
│   Scheduler     │     │   Scheduler     │
└───────┬─────────┘     └───────┬─────────┘
        │                       │
        └───────────┬───────────┘
                    ▼
        ┌───────────────────────┐
        │   共享就绪任务列表     │
        │  (带核心亲和性标记)    │
        └───────────────────────┘
                    │
        ┌───────────┴───────────┐
        ▼                       ▼
   ┌─────────┐             ┌─────────┐
   │ Task A  │             │ Task B  │
   │ Core: 0 │             │ Core: 1 │
   └─────────┘             └─────────┘
   ┌─────────┐
   │ Task C  │  ← tskNO_AFFINITY: 两个核心都可以运行
   │Core: ANY│
   └─────────┘
```

### 实际应用示例

```c
// WiFi 任务 - 绑定到核心 0（系统默认）
// CONFIG_ESP_WIFI_TASK_PINNED_TO_CORE_0=y

// 飞控关键任务 - 应绑定到核心 1（避免 WiFi 干扰）
xTaskCreateStaticPinnedToCore(
    stabilizerTask,
    "STABILIZER",
    stackDepth,
    NULL,
    STABILIZER_TASK_PRI,
    stackBuffer,
    &taskBuffer,
    1  // APP_CPU - 核心 1
);

// 日志任务 - 不绑定（任意核心）
xTaskCreateStaticPinnedToCore(
    logTask,
    "LOG",
    stackDepth,
    NULL,
    LOG_TASK_PRI,
    stackBuffer,
    &taskBuffer,
    tskNO_AFFINITY
);
```

### 双核注意事项

```c
// ⚠️ 共享数据必须加锁！
static portMUX_TYPE myMutex = portMUX_INITIALIZER_UNLOCKED;

void accessSharedData() {
    taskENTER_CRITICAL(&myMutex);  // 自旋锁
    // 访问共享数据...
    taskEXIT_CRITICAL(&myMutex);
}

// ⚠️ 某些外设只能从特定核心访问
// 例如：WiFi 相关操作只能在核心 0
```

## ESP32 SMP vs HPM AMP 架构对比

### 核心差异：SMP vs AMP

```
┌────────────────────────────────────────────────────────────────────┐
│  SMP (Symmetric Multi-Processing) - ESP32                          │
│  对称多处理                                                         │
│                                                                    │
│  • 两个核心地位平等                                                  │
│  • 运行同一个 OS 实例                                                │
│  • 任务可在任意核心运行                                              │
│  • 简单易用                                                         │
└────────────────────────────────────────────────────────────────────┘

┌────────────────────────────────────────────────────────────────────┐
│  AMP (Asymmetric Multi-Processing) - HPM6750                       │
│  非对称多处理                                                       │
│                                                                    │
│  • 两个核心独立运行                                                  │
│  • 各自运行独立 OS（甚至不同 OS）                                    │
│  • 任务固定在特定核心                                                │
│  • 灵活但复杂                                                       │
└────────────────────────────────────────────────────────────────────┘
```

### 架构对比图

```
┌─────────────────────────────────────────────────────────────┐
│                    ESP32-S3 (SMP 架构)                       │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│   Core 0 ◄────────── 共享内存 ──────────► Core 1            │
│     │                   │                    │              │
│     └───────────────────┴────────────────────┘              │
│                         │                                   │
│              ┌──────────┴──────────┐                        │
│              │  统一的 FreeRTOS    │                        │
│              │  (一个OS实例)       │                        │
│              └─────────────────────┘                        │
└─────────────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────────────┐
│                    HPM6750 (AMP 架构)                        │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│   Core 0                                    Core 1          │
│   ┌─────────┐                              ┌─────────┐      │
│   │ OS 实例1│     ┌────────────────┐       │ OS 实例2│      │
│   │(FreeRTOS)│◄───►│  IPC 通信机制  │◄─────►│(FreeRTOS)│     │
│   └─────────┘     │  - MBX 邮箱    │       └─────────┘      │
│                   │  - 共享内存    │                        │
│                   │  - 软件中断    │                        │
│                   └────────────────┘                        │
└─────────────────────────────────────────────────────────────┘
```

### 详细对比

| 维度           | ESP32-S3 (SMP)     | HPM6750 (AMP)      |
| -------------- | ------------------ | ------------------ |
| **编程复杂度** | ⭐ 简单             | ⭐⭐⭐ 复杂           |
| **实时性保证** | ⭐⭐ 中等            | ⭐⭐⭐ 强             |
| **核心隔离**   | ❌ 无               | ✅ 完全隔离         |
| **灵活性**     | ⭐⭐ 中等            | ⭐⭐⭐ 高             |
| **资源利用率** | ⭐⭐⭐ 高             | ⭐⭐ 中等            |
| **故障隔离**   | ❌ 差               | ✅ 好               |
| **异构 OS**    | ❌ 不支持           | ✅ 支持             |
| **OS 实例**    | 1个（共享）        | 2个（独立）        |
| **内存模型**   | 完全共享           | 各自独立 + 共享区  |
| **任务调度**   | 统一调度器         | 各自独立调度       |
| **核间通信**   | 直接访问共享变量   | 需要 IPC 机制      |
| **编程模型**   | 像单核一样（加锁） | 分布式（消息传递） |
| **固件**       | 1份                | 2份（每核一份）    |

### HPM AMP 的优势

#### 1. 硬实时隔离
```
┌─────────────────────────────────────────────────┐
│                   HPM6750                        │
├──────────────────────┬──────────────────────────┤
│      Core 0          │        Core 1            │
│  ┌────────────────┐  │  ┌────────────────────┐  │
│  │   Linux/RTOS   │  │  │   裸机/硬实时RTOS  │  │
│  │   WiFi/BT/GUI  │  │  │   电机控制/FOC     │  │
│  │   非实时任务   │  │  │   < 1μs 响应      │  │
│  └────────────────┘  │  └────────────────────┘  │
│                      │                          │
│   Core 0 卡死       │   Core 1 继续运行 ✅    │
└──────────────────────┴──────────────────────────┘
```

**ESP32 做不到**：如果 WiFi 任务阻塞，可能影响飞控任务。

#### 2. 混合 OS 运行
```c
// HPM 可以这样：
// Core 0: 运行 Linux（复杂 UI、网络）
// Core 1: 运行裸机程序（微秒级电机控制）

// ESP32 不行：
// 两个核心必须运行同一个 FreeRTOS
```

#### 3. 安全隔离（功能安全）
```
汽车/工业应用需要：
┌────────────────┐    ┌────────────────┐
│    Core 0      │    │    Core 1      │
│   非安全功能    │ 隔离 │   安全功能     │
│   娱乐系统     │◄───►│   刹车控制     │
│   ASIL-QM     │    │   ASIL-D       │
└────────────────┘    └────────────────┘
        │                    │
        └────── 互不干扰 ─────┘
```

#### 4. 独立看门狗
```c
// HPM: 每个核心独立看门狗
// Core 0 崩溃 → Core 0 复位 → Core 1 继续运行

// ESP32: 共享系统
// 一个核心问题可能导致整体复位
```

### ESP32 SMP 的优势

#### 1. 编程简单
```c
// ESP32: 就像写单核程序
void myTask(void *arg) {
    while(1) {
        doWork();  // 自动调度到任意核心
        vTaskDelay(10);
    }
}
xTaskCreate(myTask, "task", 4096, NULL, 5, NULL);

// HPM: 需要考虑核间通信
void core0_task(void *arg) {
    // 准备数据
    shared_mem->data = value;
    __DSB();  // 内存屏障
    mbx_send(CORE1, MSG_READY);  // 通知核心1
    mbx_wait(CORE0, MSG_DONE);   // 等待响应
}
```

#### 2. 负载均衡
```
ESP32 自动负载均衡：
┌─────────┐     ┌─────────┐
│ Core 0  │     │ Core 1  │
│  40%    │     │  40%    │  ← 自动分配
└─────────┘     └─────────┘

HPM 需手动规划：
┌─────────┐     ┌─────────┐
│ Core 0  │     │ Core 1  │
│  80%    │     │  20%    │  ← 分配不当就浪费
└─────────┘     └─────────┘
```

#### 3. 资源共享高效
```c
// ESP32: 直接共享，零拷贝
TaskA (Core0): buffer[0] = data;
TaskB (Core1): use(buffer[0]);  // 直接访问

// HPM: 需要拷贝到共享区
Core0: memcpy(shared_region, data, size);
       send_ipc_notification();
Core1: wait_ipc_notification();
       memcpy(local_data, shared_region, size);
```

#### 4. 调试简单
```
ESP32:
• 单一固件
• 统一日志
• 一个调试会话

HPM:
• 两份固件
• 两套日志
• 需要同时调试两个核心
```

### ESP32-S3 核间同步方式

虽然不需要"通信线程"，但仍需要同步机制：

```c
// 1. 自旋锁（短临界区）
portMUX_TYPE mux = portMUX_INITIALIZER_UNLOCKED;
taskENTER_CRITICAL(&mux);
// 临界区代码
taskEXIT_CRITICAL(&mux);

// 2. 互斥锁（可阻塞）
SemaphoreHandle_t mutex = xSemaphoreCreateMutex();
xSemaphoreTake(mutex, portMAX_DELAY);
// 临界区代码
xSemaphoreGive(mutex);

// 3. 队列（线程安全的数据传递）
QueueHandle_t queue = xQueueCreate(10, sizeof(int));
xQueueSend(queue, &data, portMAX_DELAY);
xQueueReceive(queue, &data, portMAX_DELAY);

// 4. 任务通知（轻量级信号）
xTaskNotify(taskHandle, 0, eNoAction);
```

### 适用场景

| 场景             | 推荐架构  | 原因                           |
| ---------------- | --------- | ------------------------------ |
| **无人机飞控**   | ESP32 SMP | 任务耦合紧密，需要快速数据共享 |
| **工业伺服驱动** | HPM AMP   | 需要微秒级硬实时隔离           |
| **智能家居**     | ESP32 SMP | 简单易开发                     |
| **汽车 ECU**     | HPM AMP   | 功能安全要求                   |
| **机器人控制**   | 视情况    | 简单机器人用 SMP，高端用 AMP   |

### 类比理解

| 架构             | 类比                                             |
| ---------------- | ------------------------------------------------ |
| **ESP32-S3 SMP** | 同一个房间里的两个人，共用一张桌子，说话就能沟通 |
| **HPM6750 AMP**  | 两个房间里的两个人，需要通过门缝传纸条沟通       |

### 为什么飞控项目用 ESP32 够了？

```
ESP-Drone 的任务：
┌────────────────────────────────────────────┐
│  stabilizer (1000Hz)  ← 最关键，但不需要 <1μs │
│  estimator (1000Hz)                         │
│  commander (100Hz)                          │
│  WiFi (异步)                                │
└────────────────────────────────────────────┘

• 1000Hz = 1ms 周期，ESP32 FreeRTOS 完全能保证
• WiFi 虽然会抢占，但影响可接受（几十微秒）
• 不需要 HPM 那种微秒级隔离
```

## 总结

| 问题                           | 答案                                          |
| ------------------------------ | --------------------------------------------- |
| **ESP-Drone 用到双核了吗？**   | 配置启用，但未显式绑定核心，依赖自动调度      |
| **两个函数区别**               | `PinnedToCore` 多了核心指定参数               |
| **tskNO_AFFINITY**             | 任务可在任意核心运行，调度器自动分配          |
| **ESP32 双核原理**             | ESP-IDF 修改了 FreeRTOS 内核，增加了 SMP 支持 |
| **ESP32 需要核间通信线程吗？** | **不需要**，SMP 架构共享内存，用锁保护即可    |
| **HPM vs ESP32**               | AMP 更强但复杂，SMP 简单够用                  |
| **飞控项目选哪个？**           | ESP32 SMP 完全足够                            |
