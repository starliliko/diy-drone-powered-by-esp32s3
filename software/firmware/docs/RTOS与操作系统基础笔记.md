# RTOS 与操作系统基础笔记

> 面向 ESP32/FreeRTOS/飞控项目的简明 OS 知识整理

---

## 1. FreeRTOS 任务、本质上是什么？

- **FreeRTOS 任务 ≈ 单进程里的线程**
  - 所有任务共享同一份地址空间：代码段、全局变量、堆、外设寄存器。
  - 每个任务有自己的栈和任务控制块（TCB）。
  - 调度器在任务间做上下文切换：保存/恢复寄存器和栈指针。
- **不是“进程”**
  - MCU 上一般没有 MMU，也就没有“多个独立虚拟地址空间”的进程概念。
  - 没有强隔离：一个任务写坏内存，整个系统都可能崩。

---

## 2. 阻塞 / 就绪 / 运行：任务状态

- **运行态（Running）**：CPU 当前正在执行该任务。
- **就绪态（Ready）**：任务已经具备运行条件，但暂时没轮到；只要调度器选中，就能立刻跑。
- **阻塞态（Blocked）**：任务在“等某件事”，当前根本不需要 CPU：
  - 等时间：`vTaskDelay()` / `vTaskDelayUntil()`；
  - 等队列：`xQueueReceive()`；
  - 等信号量/事件：`xSemaphoreTake()` / `xEventGroupWaitBits()`。

**“阻塞” = 任务主动挂起自己、让出 CPU，等待条件满足再被内核唤醒。**

---

## 3. 优先级与饥饿（Starvation）

- FreeRTOS 是**抢占式、基于优先级**的调度：
  - 只要有更高优先级的就绪任务，低优先级任务不会运行。
  - 同优先级任务在开启时间片的情况下轮流运行。
- **饥饿的典型场景**：
  - 高优先级任务里没有阻塞点（不 `delay`、不等队列），而是 `while(1)` 忙等，一直就绪/运行；
  - 结果：低优先级任务几乎或永远得不到 CPU。
- 设计建议：
  - 高优先级任务必须有阻塞点（周期性 `delay` 或阻塞在队列/信号量上）；
  - 不在高优先级任务中长时间忙循环；
  - 把耗时操作拆成小步，或扔给低优先级任务/工作队列。

---

## 4. 中断与任务：谁干什么

- **中断（ISR）**
  - 硬件事件到来时打断当前任务，执行一小段中断服务函数；
  - 要求：尽量短小，避免复杂逻辑和阻塞。
- **任务（Task）**
  - 适合做相对复杂、可以阻塞/延时的工作；
  - 常见模式：ISR 里只做“发通知”，真正处理放在任务里。
- 在本工程中的实践：
  - 使用队列/事件/`workerSchedule()` 将重活从中断/定时器/回调中移出，交给普通任务执行。

## 5. workerLoop：轻量级工作队列模式

文件：`components/core/crazyflie/modules/src/worker.c`

```c
void workerLoop()
{
  struct worker_work work;

  if (!workerQueue)
    return;

  while (1)
  {
    xQueueReceive(workerQueue, &work, portMAX_DELAY);

    if (work.function)
      work.function(work.arg);
  }
}
```

- **workerQueue**：`xQueueHandle`，队列元素是 `{ function, arg }` 结构体。
- **workerLoop**：一个专职任务，从队列中取出工作项并顺序执行。
- **workerSchedule(fn, arg)**：把“工作”（函数指针 + 参数）排队，稍后由 `workerLoop` 执行。

### 5.1 同步 vs 异步（在 worker 模式下）

- 同步调用：
  ```c
  logRunBlock(&logBlocks[i]);  // 立即执行，直到完成才返回
  ```
- 异步调用：
  ```c
  workerSchedule(logRunBlock, &logBlocks[i]); // 立刻返回，稍后在 workerLoop 中执行
  ```
- 作用：
  - 调用者不被“重活”阻塞，可以尽快返回（尤其是中断回调、定时器回调、通信回调）；
  - 所有异步工作在同一任务上下文串行执行，减少加锁复杂度，复用同一个栈。

### 5.2 在本项目中的实际使用

- 日志系统 `log.c`：
  - 定时器回调中调用 `workerSchedule(logRunBlock, ...)`；
  - 避免在 FreeRTOS timer 任务里做大量打包/发送工作。
- `crtp_localization_service.c`：
  - 收到特定 CRTP 包后，将持久化操作排队到 worker；
  - 避免在通信处理上下文里直接做存储等耗时操作。

总结：**RTOS 提供“任务 + 队列”，workerLoop 在其上实现“统一后台工作线程”的模式，用来承接各种非实时、小批量异步任务。**

## 6. PX4 与 NuttX 的关系

- **PX4**：飞控中间件 + 应用层，负责传感器融合、控制、任务管理、通信等。
- **NuttX**：类 Unix 的实时操作系统，提供任务/线程调度、文件系统、设备驱动框架、网络栈等。
- 关系类比：
  - PX4 ≈ 应用/中间件；
  - NuttX ≈ OS 内核 + 驱动层；
  - 在 Pixhawk 等主流硬件上：PX4 默认运行在 NuttX 之上。
- PX4 **不必**严格依赖 NuttX：
  - 也可以运行在 Linux/macOS/Windows（SITL）、某些纯 Linux 平台等；
  - 通过不同的“平台适配层”访问 OS 功能。

### 6.1 PX4 自己的抽象：uORB / work queue / 调度

- **uORB**：发布-订阅消息总线
  - 传感器、估计、控制、导航模块通过“话题”解耦传递数据。
- **work queue**：
  - PX4 在 OS 线程之上建的一层“工作队列”系统；
  - 不同队列（高优先级、低优先级、SPI 专用等）对应不同 worker 线程。
- **调度与回调**：
  - NuttX 的线程调度器负责底层 CPU 分时；
  - PX4 在自己的 work queue 里安排模块的执行节奏，驱动回调使用 work queue/uORB 将硬件事件带入系统。

与本工程中的 `workerLoop` 类比：PX4 的 work queue 是**大号、分级版的 worker 模式**。

## 7. MMU / MPU 与 MCU 上的 RTOS

- **MMU（Memory Management Unit）**：内存管理单元
  - 功能：
    - 虚拟地址 → 物理地址翻译；
    - 地址空间隔离（不同进程互不干扰）；
    - 支持分页、换页（虚拟内存）。
  - 典型存在于 PC/手机等带虚拟内存的系统。
- **MPU（Memory Protection Unit）**：内存保护单元
  - 只做区域保护：某段内存可读/可写/可执行；
  - 无地址翻译，不提供完整虚拟内存。
- **大部分 MCU（Cortex-M、ESP32 等）没有 MMU**：
  - 没有多进程虚拟内存，每个任务直接用物理地址；
  - RTOS（如 FreeRTOS）提供的是“单地址空间、多任务”的模型。

因此：在 MCU + FreeRTOS 这种组合下，**任务永远更像“线程”而不是“进程”**，不管是单核还是多核。

---

## 8. PX4 在 ESP32-S3 / 多核上的要点

- ESP32-S3 是双核 Xtensa LX7，理论上可以跑 NuttX + PX4。
- 实际移植中，往往：
  - NuttX 在 ESP32-S3 上多核支持尚不完全成熟；
  - PX4 的很多核心设计假设“单核 RTOS”，要安全利用多核需要大量改造（任务划分、锁、实时性评估）。
- 因此现有一些移植案例：
  - 能在 ESP32-S3 上运行 PX4，但**通常以单核模式使用**（只开一核跑 PX4）。

## 9. 任务调度基础概念

### 9.1 任务状态与流转

- 状态：运行（Running）、就绪（Ready）、阻塞（Blocked）、挂起（Suspended）。
- 典型流转：
  - 创建 → 就绪 →（被调度器选中）→ 运行；
  - 运行 → 调用 `vTaskDelay` / 等队列 / 等信号量 → 阻塞；
  - 阻塞 → 时间到 / 事件到 → 回到就绪；
  - 任意时刻只能有一个任务处于“运行”状态（单核）。

### 9.2 抢占式调度与优先级

- FreeRTOS 使用**固定优先级 + 抢占式调度**：
  - 有更高优先级任务变为就绪时，立刻打断当前低优先级任务；
  - 最高优先级的就绪任务总是先运行。
- 设计时需要：
  - 给关键控制/传感器任务更高优先级；
  - 确保高优先级任务有阻塞点，避免饿死低优先级。

### 9.3 时间片与同优先级任务

- 同一优先级有多个就绪任务时，可启用**时间片轮转（Time Slicing）**：
  - 每个任务在一个时间片内运行，时间片到期后切换到同优先级的下一个任务；
  - 由 `configUSE_TIME_SLICING` 控制是否开启。
- 作用：防止同优先级下某个任务长期占用 CPU。

### 9.4 系统 tick 与调度时机

- SysTick/tick 中断周期性触发（如 1000Hz）：
  - 驱动 `vTaskDelay` / 软件定时器等“时间相关”的唤醒；
  - 为时间片轮转提供时基；
  - 在合适时机触发一次调度决策。
- 其他调度触发：
  - 任务从阻塞变就绪（队列有数据、信号量释放等）；
  - 显式调用 `taskYIELD`/`portYIELD` 让出 CPU。

### 9.5 上下文切换（Context Switch）

- 内容：保存当前任务的寄存器、栈指针、程序计数器等，恢复目标任务的对应值。
- 特点：
  - 有一定时间开销，任务数量和切换频率要适度；
  - 在 FreeRTOS 内核/汇编中自动完成，应用层只需理解“切换时机”和“代价”。

### 9.6 调度问题：饥饿、优先级反转、抖动

- 饥饿（Starvation）：低优先级长期得不到 CPU（例如高优先级任务从不阻塞）。
- 优先级反转：低优先级任务持有资源，高优先级任务等待资源，中优先级任务抢占低优先级 → 高优先级间接被“压低”。
  - 解决：使用带优先级继承的 Mutex。
- 抖动（Jitter）：周期任务实际执行时间点相对理想周期点的偏移，用 `vTaskDelayUntil` 等方式可减小抖动。

### 9.7 单核 vs 多核

- 单核：任一时刻只有一个任务运行，调度器选择“哪一个”上 CPU。
- 多核：同时有多个任务在不同核上运行，需要：
  - 任务亲和性（是否绑核）；
  - 跨核唤醒/通知机制；
  - 更严格的锁/同步，避免跨核竞态。
- 对当前 ESP32/FreeRTOS 飞控项目：
  - 通常采用“将关键实时任务固定在指定核 + 把后台任务安排到另一核”的策略；
  - 思想上仍是“优先级 + 抢占”模型，多核只是“多份执行资源”。

## 10. 一套适合当前阶段的 OS 学习清单

结合本工程，短期推荐重点掌握：

1. **任务模型与状态**
   - 运行 / 就绪 / 阻塞 / 挂起；
   - 抢占式优先级调度；
   - 为什么高优先级任务必须阻塞。

2. **同步与通信原语**
   - 队列（Queue）、信号量（Semaphore）、互斥量（Mutex）、事件组（Event Group）；
   - 它们的阻塞语义（等待数据/等待资源）。

3. **中断与任务交互**
   - ISR 简短处理 + 通过队列/信号量/worker 将重活交给任务。

4. **work queue / worker 模式**
   - 单 worker + 队列（本工程的 `workerLoop`）；
   - 多 work queue + worker（PX4）。

5. **内存与隔离的现实**
   - MCU 上无 MMU，所有任务共享内存；
   - 靠代码规范和少量 MPU 做“软约束”，而非硬隔离。
