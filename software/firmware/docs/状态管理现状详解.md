# 状态管理现状详解（基于当前代码）

本文档基于当前工程实现，目标是把“飞行状态由谁维护、怎么流转、在哪些地方会变化”讲清楚，方便调试、二次开发和参数联调。

## 1. 总体结论

当前架构已经形成单一状态真相源：

- 飞行器状态真相源：vehicle_state 模块（VehicleState 结构）
- 控制输入真相源：commander 当前生效 setpoint + 优先级
- 系统层兼容接口：systemIsArmed() 只读代理到 vehicleIsArmed()

一句话理解：

- 谁在飞（isArmed/isFlying/flightPhase/failsafe）由 vehicle_state 负责
- 谁在控（CRTP/REMOTE/SBUS）由 commander 负责
- 控制语义转换（RPYT -> 各模式控制量）统一走 commanderApplyFlightMode()

## 2. 模块职责划分

### 2.1 vehicle_state（状态机核心）

职责：

- 解锁/上锁状态机（Arming）
- 飞行模式状态（FlightMode）
- 飞行阶段状态（FlightPhase）
- 失控保护状态（Failsafe）

核心数据结构：

- VehicleState（包含 armingState、flightMode、flightPhase、failsafeState、isArmed、isFlying 等）

线程安全：

- 使用 stateMutex 保护状态更新
- vehicleStateUpdate() 在稳定器主循环中周期调用

### 2.2 commander（输入仲裁与语义统一）

职责：

- 收集并仲裁多输入源 setpoint（CRTP/REMOTE/SBUS）
- 优先级机制 + 锁定时间防抖
- 控制源超时检测与故障切换
- setpoint 安全钳位与合法性约束

关键机制：

- 优先级：SBUS(3) > REMOTE(2) > CRTP(1) > DISABLE(0)
- 超时：500ms 进入“保平安”姿态，2000ms 进入 shutdown 空命令
- 语义统一：所有 RPYT 输入进入控制器前都调用 commanderApplyFlightMode()

### 2.3 extrx（SBUS 输入路径）

职责：

- 读取 SBUS 帧
- 处理 arm 开关与模式开关
- 生成 setpoint 并提交 commander

当前实现特点：

- 解锁判断依赖 vehicleIsArmed() 真值
- armed 日志使用 armedLog 镜像，不再作为控制逻辑真值
- 信号丢失时会强制 disarm 并停止继续推送有效控制输入

### 2.4 system（兼容层）

职责：

- 系统初始化与测试
- 历史接口兼容

当前实现特点：

- systemIsArmed() 直接代理 vehicleIsArmed()
- 不再持有 armed 影子真值

## 3. 四类状态的定义与触发条件

### 3.1 解锁状态 ArmingState

状态集合：

- INIT
- STANDBY
- ARMED
- STANDBY_ERROR
- SHUTDOWN
- IN_AIR_RESTORE

主路径：

1. INIT：等待传感器校准完成
2. STANDBY：允许解锁
3. ARMED：已解锁，可输出动力
4. STANDBY_ERROR：紧急停机后进入错误待机

关键 API：

- vehicleArm(force)
- vehicleDisarm(force)
- vehicleEmergencyStop()
- vehicleResetEmergency()

### 3.2 飞行模式 FlightMode

状态集合：

- MANUAL, STABILIZE, ALTITUDE, POSITION, MISSION, RTL, LAND, TAKEOFF, OFFBOARD

当前实际高频使用：

- STABILIZE、ALTITUDE、POSITION

模式影响点：

- commanderApplyFlightMode() 会根据 FlightMode 重写 setpoint 控制语义：
  - STABILIZE：姿态角 + thrust 直接控制
  - ALTITUDE：thrust 改写为 z 速度命令
  - POSITION/OFFBOARD：roll/pitch 改写为 body-frame xy 速度命令

参数语义说明：

- vehicle.flightMode 参数当前为只读镜像，不再允许通过 param 直接“改模式”

### 3.3 飞行阶段 FlightPhase

状态集合：

- ON_GROUND
- TAKEOFF
- IN_AIR
- LANDING
- LANDED（当前逻辑中会归并回 ON_GROUND）

判定逻辑采用“双信号”：

- 控制意图：来自 commander 当前 setpoint（takeoff/landing command）
- 物理确认：来自 stateEstimate.z 与 stateEstimate.vz

关键转换：

- ON_GROUND -> TAKEOFF：检测到起飞意图
- TAKEOFF -> IN_AIR：估计高度超过阈值（含超时兜底）
- IN_AIR -> LANDING：检测到降落意图
- LANDING -> ON_GROUND：时间阈值 + 近地低速物理确认（含超时兜底）

### 3.4 失控保护 FailsafeState

状态集合：

- NONE
- LOW_BATTERY
- CRITICAL_BATTERY
- RC_LOSS
- GCS_LOSS
- SENSOR_FAILURE
- GEOFENCE

当前已生效重点：

- RC_LOSS：已解锁且 RC 断连时触发
- RC 恢复后清除 RC_LOSS

当前未完整接入：

- 电池低电量链路在本项目里被临时禁用（避免无效电压误判）

## 4. 关键时序（从输入到状态）

### 4.1 正常飞行控制路径

1. 输入源（CRTP/REMOTE/SBUS）产生 RPYT 控制命令
2. 各输入路径在提交前调用 commanderApplyFlightMode()
3. commander 按优先级写入 setpointQueue
4. stabilizer 周期读取 setpoint 并执行控制
5. stabilizer 周期调用 vehicleStateUpdate() 维护状态机

### 4.2 SBUS 解锁与失联路径

1. SBUS 任务读帧并判定 arm 开关
2. arm 开时调用 vehicleArm(false)
3. arm 关时调用 vehicleDisarm(false)
4. 信号丢失时调用 vehicleDisarm(true) 并设置 RC 断连
5. vehicle_state 在 updateFailsafe() 中进入或退出 RC_LOSS

### 4.3 控制源故障切换路径

1. commander 记录各优先级源最近活跃时间
2. 当前优先级源超时时触发 failover
3. 选择仍在线的最高优先级源
4. 若都失联，退到 DISABLE + 安全姿态

## 5. 解锁前检查机制（runArmingChecks）

当前启用检查：

- 传感器校准完成（ARM_CHECK_SENSORS）
- 油门归零/无起飞意图（ARM_CHECK_THROTTLE_ZERO）

当前临时禁用：

- 电池检查（ARM_CHECK_BATTERY）

失败原因可通过以下接口/日志定位：

- vehicleGetArmFailReason()
- vehicleGetArmFailReasonStr()

## 6. 观测与调试建议

### 6.1 重点日志组

- vehicle：armState, flightMode, flightPhase, failsafe, isArmed, isFlying
- commander：activePri, accepted, rejected, invalid, extRxAlive/remoteAlive/crtpAlive
- extrx：ch0~ch9, thrust/roll/pitch/yaw, armed, mode

### 6.2 联调时推荐同时看

- vehicle.flightPhase + vehicle.isFlying
- stateEstimate.z + stateEstimate.vz
- commander.activePri + commander.*Alive
- extrx.armed + vehicle.isArmed（应一致）

### 6.3 常见误解

- 误解1：改 vehicle.flightMode 参数能切模式
  - 现状：该参数只读，真实切模式应走 vehicleSetFlightMode() 的调用链
- 误解2：system 层还维护独立解锁状态
  - 现状：systemIsArmed() 只代理 vehicle_state 真值

## 7. 当前边界与后续可优化点

以下不是架构缺陷，属于可持续优化项：

1. commander 的 timeoutCount 目前在每次收到包时清零，统计含义偏“当前态”，不是“累计可靠性”
2. SOURCE_RETRY_INTERVAL_MS 宏当前未被使用，建议清理或落实
3. extrx.currentMode 与 vehicle.flightMode 仍是并行变量，虽然功能正常，但可以继续收口为单真值读取
4. 失控保护中的 GCS_LOSS、SENSOR_FAILURE、电池链路可按硬件实况逐步补齐

## 8. 你在开发中应该怎么用

推荐遵循以下规则：

1. 任何新输入源，只负责产生“原始控制意图”，然后调用 commanderApplyFlightMode()
2. 不要在外部模块维护 isArmed 影子真值，统一读 vehicleIsArmed()
3. 不要直接改日志镜像变量（logFlightMode 等），只通过 vehicle_state API 改真实状态
4. 把 vehicleStateUpdate() 继续作为唯一状态机推进入口

## 9. 快速检查清单

每次改动后，至少验证以下 6 点：

1. 解锁前油门检查是否仍生效
2. SBUS arm 开关、CRTP、remote 三路切换是否不会互相“抢假状态”
3. RC 断连是否触发强制上锁 + RC_LOSS
4. TAKEOFF/LANDING 阶段转换是否同时满足意图与物理确认
5. commander 优先级与 failover 是否符合预期
6. vehicle 与 commander 日志字段是否一致、不打架
